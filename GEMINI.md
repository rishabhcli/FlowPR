# FlowPR

## Project Overview

FlowPR is an autonomous frontend QA engineer designed for developers and software teams. It tests important user flows in a live browser, captures visual evidence, diagnoses failures, prepares focused code changes, verifies fixes, and opens GitHub pull requests with proof. 

The project is built as a **Node.js monorepo** managed by `pnpm` and structured using workspaces. 

### Architecture & Key Technologies
- **Language:** TypeScript (strict mode)
- **Monorepo:** `pnpm` workspaces
- **Dashboard (`apps/dashboard`):** Next.js App Router, Tailwind CSS, shadcn/ui. Acts as the command center to start runs and view state.
- **Demo Target (`apps/demo-target`):** A seeded Next.js target application for testing and demonstration.
- **Agent Worker (`packages/agent`):** A Node.js TypeScript service with an explicit, deterministic state machine driven by Redis consumer groups.
- **Core Integrations (Sponsor Stack):**
  - **TinyFish:** Live browser automation (Agent API and Browser API).
  - **Redis:** Event bus (Redis Streams/`XREADGROUP`), autonomous run state, locks, and agent memory.
  - **InsForge:** Durable backend (system of record) for runs, artifacts, observations, patches, and PR metadata (PostgreSQL).
  - **WunderGraph:** Safe action layer exposing permitted GraphQL operations.
  - **Guild.ai:** Agent control plane (registry, versioning, permission gates, execution trace).
  - **Senso:** Knowledge grounding for product specs, QA policies, and rules.
  - **Shipables:** Used to package FlowPR as a reusable agent skill (`skills/flowpr-autonomous-frontend-qa`).
  - **Playwright:** Local repeatable E2E tests and trace artifacts.
  - **GitHub:** Pull Request creation and API.

## Building and Running

### Prerequisites
- Node.js 20+
- `pnpm` 10
- A local or remote Redis instance (e.g., `brew install redis && brew services start redis`)
- Environment variables configured in `.env` (copied from `.env.example`)

### Installation
```bash
pnpm install
```

### Running the Stack Locally
The application requires three long-running processes:

```bash
# 1. Start the Next.js Dashboard UI (http://localhost:3000)
pnpm dev:dashboard

# 2. Start the Redis consumer worker that drives the agent
pnpm worker:dev

# 3. (Optional) Start the seeded target app for testing (http://localhost:3100)
pnpm dev:demo
```

### Key Scripts & Commands
- **Typecheck:** `pnpm typecheck` (runs `tsc --noEmit` across all packages)
- **Lint:** `pnpm lint`
- **Build:** `pnpm build` (or `pnpm --filter @flowpr/dashboard build`)
- **Smoke Tests & Evaluation:**
  - `pnpm redis:smoke` (asserts streams, consumer groups, locks)
  - `pnpm guild:evaluate` (validates agent capabilities and gates)
- **Demo Loop Commands:**
  - `pnpm demo:break` (injects a visual bug into the demo target)
  - `pnpm demo:start-run` (starts a run via the API)
  - `pnpm demo:reset` (restores the demo target app)

## Development Conventions

- **Strict TypeScript:** The codebase enforces strict TypeScript rules across all packages.
- **Deterministic Agent State:** The agent worker (`packages/agent/src/state-machine.ts`) operates on a rigid, explicit state machine (e.g., `queued` -> `loading_repo` -> `running_browser_qa` -> ... -> `creating_pr` -> `done`). It avoids free-form LLM planning for core flow control.
- **Evidence-Based Actions:** Code changes and PRs should only be generated with accompanying evidence (before/after screenshots, Playwright traces, and JSON logs). 
- **Safe Action Boundaries:** The agent must not execute arbitrary code modifications or terminal commands directly in production; state updates and PR creation should route through the WunderGraph safe operations layer.
- **Testing:** New fixes generated by FlowPR must include an updated or new Playwright regression test to verify the fix locally before opening a PR.
